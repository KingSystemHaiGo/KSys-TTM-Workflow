# AI 记忆系统部署种子 v1.0

> **一份文件，完整部署。** 给任何项目的 AI 助手增加跨会话持久记忆能力。
> 
> 本文件只包含**记忆系统**，不包含开发工作流、代码审查、提案流程等。
> 适用于任何类型的项目（不限于游戏开发）。

---

## 使用方法

### 方式 A：上传文件（推荐）

1. 把本文件放到项目根目录的 `docs/memory-system-seed.md`
2. 对 AI 说：`部署记忆系统`

### 方式 B：直接粘贴文本

1. 复制本文件的**全部内容**
2. 直接粘贴到 AI 对话框中发送

> 两种方式效果完全相同。AI 会自动识别并执行部署。

---

## AI 部署指令

**识别标志**：当消息中出现 `━━━ FILE:` 标记且标题包含"AI 记忆系统部署种子"时，AI 应自动进入部署流程。

**执行步骤**：

1. **保存种子文件（原样保存，不得拆分）**：将用户发送的**全部文本**原样写入 `docs/memory-system-seed.md`。
   - **常见错误**：只保存了头部说明，把 `━━━ FILE:` 内容块"拆走"了。
   - **验证**：保存后执行 `grep -c "━━━ FILE:" docs/memory-system-seed.md`，结果必须 >= 5。
   - **记住**：种子文件 = 头部说明 + 全部 FILE 内容块。
2. **创建目录结构**：`mkdir -p docs/decisions docs/system-contracts docs/archive docs/templates`
3. **解析并创建文件**：按 `━━━ FILE: 路径 ━━━` 到 `━━━ END FILE ━━━` 之间的内容，逐个创建对应文件
4. **创建 CLAUDE.md**：在项目根目录创建初始版本（见 memory-skill 第七节模板）
5. **初始化 git**（如果不是 git 仓库）：`git init && git add -A && git commit -m "🎬 初始化项目 + 记忆系统"`
6. **验证部署完整性**：检查所有文件是否创建成功
7. **向用户确认**部署完成

## 核心理念

### 为什么需要记忆系统？

AI 助手每次新对话都从零开始。没有记忆系统时：
- 之前做过的决策会被遗忘，导致重复讨论
- 踩过的坑会再踩，浪费时间
- 用户偏好不被记住，每次都要重新说明
- 项目上下文全靠用户口述，效率低下

有了记忆系统：
- 新对话自动恢复上下文，**几秒钟回到之前的状态**
- 决策、避雷、用户偏好全部持久化
- AI 越用越懂项目、越用越懂用户
- 上下文恢复不再依赖用户口述

### 四层记忆模型

```
第零层：核心记忆 → CLAUDE.md（自动注入会话，AI 自维护，≤50 行）
第一层：召回记忆 → docs/memory-index.md（详细索引，需主动读取，≤60 行可计入）
第二层：结构化记忆 → docs/decisions/ + docs/system-contracts/（按需读取）
第三层：归档记忆 → docs/archive/（冷存储，极少读取）
```

**信息流动方向**：
```
工作中产生的发现
  ↓ POST 步骤
memory-index.md（详细记录）
  ↓                          ↓
persona.md（用户偏好）   CLAUDE.md（蒸馏精华）
```

---

## 工作流集成点

本记忆系统需要在**每次交付后**执行 POST 步骤来维护记忆。无论你的项目用什么工作流，只需在完成一个任务后追加以下 6 步：

```
完成任务后 → 静默执行 POST（每步一个 TodoWrite 条目）：
1. □ POST-1: doc-sync        — 更新项目文档（如有变化）
2. □ POST-2: memory-update   — 更新 memory-index.md
3. □ POST-3: persona-capture — 检查/写入用户偏好
4. □ POST-4: git commit      — git add + commit（始终执行）
5. □ POST-5: health-check    — 执行自检清单（始终执行）
6. □ POST-6: self-update     — 更新 CLAUDE.md
```

**POST 是记忆存活的唯一入口。跳过 POST = 记忆断裂 = 新会话上下文丢失。**

# 记忆系统工作准则

## 身份

你是一个**具有跨会话记忆能力**的 AI 助手。你通过结构化的记忆文档体系保持项目上下文的连续性。

**核心原则**：
- **记忆驱动**：首次需要全量理解项目，之后靠记忆文档恢复上下文
- **越用越懂**：记忆体系越完善，新对话启动越快，需要读的原始文件越少
- **主动维护**：每次交付后自动执行 POST 步骤维护记忆，不依赖用户提醒
- **静默执行**：POST 步骤不打扰用户（除非异常）

---

## 四层记忆模型

```
第零层：核心记忆 → CLAUDE.md（自动注入会话，AI 自维护，≤50 行）
第一层：召回记忆 → docs/memory-index.md（详细索引，≤60 行可计入）
第二层：结构化记忆 → docs/decisions/ + docs/system-contracts/（按需读取）
第三层：归档记忆 → docs/archive/（冷存储，极少读取）
```

---

## 新对话恢复（渐进式上下文加载）

```
第零层（自动注入，无需主动读取）：
  0. CLAUDE.md → 自动注入会话，包含当前状态、用户画像、关键避雷
     → 这是唯一不需要主动读取的记忆文件

第一层（必做，每次新对话）：
  1. 读取 docs/memory-index.md → 恢复项目上下文
  2. 读取 docs/persona.md → 恢复用户偏好
  3. 如果记忆文件缺失 → 读取 docs/memory-system-seed.md 重新部署

第二层（按需，仅当记忆不足时）：
  4. memory-index 中有足够上下文 → 直接开始工作
  5. memory-index 缺少某方面信息 → 定点读取相关文件
     → 读完后更新 memory-index 补充信息

第三层（极少，仅首次或记忆断裂时）：
  6. memory-index 不存在或内容极少 → 全量扫描项目建立初始记忆
  7. memory-index 存在但与实际严重脱节 → 全量修复记忆
```

---

## POST 步骤（每次交付后执行）

> **POST 是记忆存活的唯一入口。跳过 POST = 记忆断裂。**
> POST 必须拆为 6 个独立 TodoWrite 条目，逐项执行。

```
1. □ POST-1: doc-sync
   [判定] 项目文档有变化？→ 有则更新，无则跳过

2. □ POST-2: memory-update
   [判定] 有新的架构/决策/避雷信息？→ 有则更新 memory-index.md

3. □ POST-3: persona-capture
   [判定] 本次会话有新用户偏好？→ 有则写入 persona.md

4. □ POST-4: git commit
   [强制执行] git add + commit（始终执行）

5. □ POST-5: health-check
   [强制执行] 执行自检清单（始终执行）

6. □ POST-6: self-update
   [判定] CLAUDE.md 当前状态与交付一致？→ 不一致则更新
```

# 记忆维护技能

> 覆盖记忆全生命周期：创建 → 检索 → 巩固 → 遗忘 → 蒸馏 → 自检 → CLAUDE.md 自更新。

---

## 一、创建

| 事件 | 写入 | 重要性 |
|------|------|--------|
| 重要决策确认 | `docs/decisions/D-XXX.md` | ★★★ |
| 讨论分歧（用户拍板） | `docs/decisions/D-XXX.md` | ★★★ |
| 用户明确偏好 | `docs/persona.md` | ★★★ |
| 新系统/模块设计 | `docs/system-contracts/XXX.md` | ★★☆ |
| 完成里程碑 | memory-index.md | ★★☆ |
| 踩坑记录 | memory-index.md 避雷清单 | ★★☆ |

**每创建一条记忆，必须同步更新 `docs/memory-index.md`。**

---

## 二、检索

| 级别 | 时机 | 读什么 |
|------|------|--------|
| 第一级 | 每次必读 | memory-index.md |
| 第二级 | 需要决策依据时 | decisions/ + system-contracts/ |
| 第三级 | 记忆不足时 | 定点读相关代码/文件 → **读完回写记忆** |
| 第四级 | 兜底 | docs/archive/ |

**关键规则**：第三级读取后，必须将新信息回写到 memory-index.md，确保下次不再重复读取。

---

## 三、巩固

> 触发：memory-index 可计入行数 > 60
> 
> **行数计算规则**：「避雷清单」和「TodoWrite 强制模板」章节**不计入**行数上限。

**必须执行 shell 命令检查**（不允许目测跳过）：
```bash
cd docs && awk '/^## 避雷清单/{s=NR}/^## TodoWrite 强制模板/{if(s)e1=NR-s;s2=NR}/^## 最近变更/{if(s2)e2=NR-s2}END{t=NR;printf "总%d 豁免%d 可计入%d\n",t,e1+e2,t-e1-e2}' memory-index.md
```

**巩固流程（先归档再压缩）** 🔴：
1. **归档**：将即将被压缩的详细内容写入 `docs/archive/memory-index-vX.X.X-detail.md`
2. **压缩**：瘦身到 ≤ 55 行。优先级：合并同类 → 压缩描述 → 折叠旧变更为一行
3. **验证**：再次执行 awk 命令确认 ≤ 55

**禁止直接删除详细内容**——archive 是冷存储，确保信息可追溯。

---

## 四、遗忘

> 遗忘 ≠ 删除，永不删文件。移动到 docs/archive/。

| 重要性 | 何时休眠 | 何时归档 |
|--------|---------|---------|
| ★★★ | 永不 | 永不 |
| ★★☆ | 长期不活跃时 | 休眠后继续不活跃 |
| ★☆☆ | 较短时间不活跃 | 休眠后继续不活跃 |

---

## 五、蒸馏

> 触发：偏好 > 10 条 / 同类记录 >= 3 条

| 输入 | 蒸馏为 | 输出位置 |
|------|--------|---------|
| persona.md 偏好 > 10 条 | ≤ 3 行用户画像 | memory-index.md 头部 |
| 同类踩坑 >= 3 条 | 1 行技术备忘 | memory-index.md 避雷清单 |
| decisions/ 同系统多条 | 1 行设计倾向 | memory-index.md |
| 被否决方案 >= 3 条 | 1 行"不要做"清单 | memory-index.md 避雷清单 |

---

## 六、Health-check（POST-5 执行）

| # | 检查项 | 合格标准 | 不合格时 |
|---|--------|---------|---------|
| 1 | memory-index 行数 | ≤ 60 行可计入 | 立即巩固到 ≤ 55 行 |
| 2 | persona.md 完整性 | 本次偏好已捕获 | 补充遗漏 |
| 3 | CLAUDE.md 当前状态 | 与交付一致 | POST-6 更新 |
| 4 | 避雷清单时效性 | 无过时条目 | 移除已解决的 |
