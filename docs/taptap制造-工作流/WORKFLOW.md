# TapTap 制造工作流：从想法到上架

> 一套面向复杂游戏项目的 AI 协作开发工作流。
> 前半程追求速度——最快出原型；后半程追求质量——系统化地打磨、不失控。

---

## 快速开始

**如果你只想赶紧用起来，看 [DEPLOY.md](DEPLOY.md)——三步部署，两分钟搞定。**

部署后 AI 会自动执行下面描述的全部流程，你不需要手动操作。
以下内容是给想理解原理的人看的。

## 架构总览

```
┌─────────────────────────────────────────────┐
│  智能体准则（CLAUDE.md）— 永远在线的大脑     │
│  · 阶段感知：自动判断项目处于哪个阶段        │
│  · 自动行为：不需要用户开口就执行的操作       │
│  · 记忆管理：每次对话自动加载/更新项目记忆    │
│  · 自我反思：每次交付后内部质量检查           │
├─────────────────────────────────────────────┤
│  Skill 矩阵 — 按需加载的专业能力             │
│  · workflow-auto：四阶段流程的详细行为        │
│  · self-reflect：交付后复盘 + 记忆维护       │
├─────────────────────────────────────────────┤
│  项目记忆 — AI 自动创建和维护的文档           │
│  · memory-index.md：记忆索引（轻量，常驻）    │
│  · decision-log.md：决策账本（按需读取）      │
│  · system-contracts/：系统契约（按需读取）    │
│  · milestones.md：里程碑记录                 │
└─────────────────────────────────────────────┘
```

用户唯一需要做的：说想做什么游戏，然后在 AI 给的选项里选一选。

---

## 这套工作流解决什么问题

简单游戏一轮对话就能搞定，不需要工作流。

但当你的项目有 5+ 个互相影响的系统（移动、战斗、UI、存档、音效……），跨越 10+ 轮对话，代码超过 1000 行时，你会遇到三个问题：

1. **决策衰减**：第 1 轮定的架构，第 8 轮 AI 已经忘了，写出风格矛盾的代码
2. **隐式耦合**：跳跃系统和受击系统单独都对，但"跳跃中受击"没人定义过
3. **回溯代价**：第 10 轮发现第 4 轮的方向不对，但 5-9 轮全建在上面

这套工作流用四个阶段 + 四个文档模板来管理这些复杂性。

---

## 全局流程

```
阶段 1：原型冲刺          ← 目标：最快出可玩原型
    ↓
阶段 2：功能搭建          ← 目标：系统逐个就位，接口清晰
    ↓
阶段 3：打磨优化          ← 目标：手感、美术、音效、平衡性
    ↓
阶段 4：发布上架          ← 目标：素材齐全，测试通过，提交 TapTap
```

每个阶段有明确的**进入条件**和**产出物**，不满足进入条件不要强行推进。

---

## 阶段 1：原型冲刺

### 目标
用最短时间做出一个**能玩的、丑的、但核心循环完整的**原型。

### 时机
你有一个想法，但还没写代码。

### 做法

**第一步：填写游戏设计简报**

打开 `templates/game-brief.md`，填写三个必填项：

| 必填项 | 回答什么 | 示例 |
|--------|---------|------|
| 核心动词 | 玩家主要在做什么 | "切"（水果忍者）、"跳"（马里奥）、"搭"（俄罗斯方块） |
| 胜负条件 | 怎么赢/怎么输 | "三条命用完结束，分数越高越好" |
| 一句话描述 | 说给朋友听的一句话 | "用手指切飞出来的水果，别切到炸弹" |

其余项（美术风格、目标平台、参考游戏）选填，想到就写，想不到先空着。

**第二步：把简报交给 AI**

```
"我要做一个游戏，设计简报在 docs/game-brief.md，请读完后：
1. 选合适的脚手架
2. 做一个最小可玩原型
3. 不用管美术，色块就行
4. 交付后附预期表现清单"
```

**第三步：玩一玩，调手感**

原型出来后，不要急着加功能。先反复玩几次，回答这些问题：

- 核心操作（移动/跳跃/攻击）是否顺手？
- 节奏是太快还是太慢？
- 最基本的"好玩"感觉有没有？

把感受告诉 AI，调整参数。这个阶段的目标是**手感对了**，不是功能全了。

### 产出物
- [x] `docs/game-brief.md`（已填写）
- [x] `scripts/main.lua`（可运行的最小原型）
- [x] 核心手感确认（你玩过并且觉得"基本操作是舒服的"）

### 进入阶段 2 的条件
> 核心循环能跑，核心操作手感你满意。

---

## 阶段 2：功能搭建

### 目标
在原型基础上，逐个添加游戏系统，同时管控复杂度。

### 这个阶段最容易出问题

阶段 1 很快，阶段 3 很直觉，**阶段 2 是整个项目成败的关键**。因为：

- 系统之间开始产生依赖
- 代码量快速膨胀
- 早期的随意决策开始产生后果

### 做法

**每加一个系统前，做三件事：**

#### 1. 写系统契约（`templates/system-contract.md`）

一页纸，回答四个问题：

```
这个系统需要什么输入？（从哪些系统读数据）
这个系统产生什么输出？（改变什么、通知什么）
和现有系统有哪些交叉场景？（列出来，每个写一行处理方式）
边界情况？（空值、极端值、同时触发）
```

**交叉场景是最重要的部分。** 比如你要加"受击系统"，必须回答：

- 跳跃中被击中 → 中断跳跃？还是空中受击？
- 攻击动画中被击中 → 中断攻击？还是霸体？
- 无敌帧期间被击中 → 忽略？

这些不提前想清楚，后面就是 bug。

#### 2. 记决策（`templates/decision-log.md`）

每个非显而易见的决策都记一条：

```
## D-005: 受击硬直时长
- 选择：0.3 秒硬直
- 理由：太短感觉不到打击感，太长影响操作流畅度
- 影响：连招系统需要考虑硬直取消
- 边界：如果后续加格挡系统，需重新评估
```

**记录的关键不是"选了什么"，而是"为什么选"和"什么条件下需要改"。**

#### 3. 告诉 AI 上下文

```
"接下来加受击系统。
- 系统契约在 docs/system-contracts/damage.md
- 决策记录在 docs/decision-log.md（请读最新的 3 条）
- 当前里程碑状态在 docs/milestones.md

请先读这些文件，然后开始实现。"
```

**每加完一个系统后，做两件事：**

#### 4. 验收 + 标里程碑（`templates/milestones.md`）

用预期清单模式验收（详见《预期清单驱动模式》），通过后记录里程碑：

```
## M-003: 角色战斗系统
- 包含：攻击、受击、无敌帧、击退
- 验收状态：通过
- 已知限制：无连招、无远程攻击
- 回退点：commit abc1234
```

#### 5. 代码膨胀检查

| 当前总行数 | 行动 |
|-----------|------|
| < 800 行 | 继续 |
| 800-1200 行 | 检查是否有可提取的独立模块 |
| > 1200 行 | **必须拆分**，和 AI 讨论模块化方案 |

### 产出物
- [x] `docs/system-contracts/*.md`（每个系统一份契约）
- [x] `docs/decision-log.md`（持续更新）
- [x] `docs/milestones.md`（每个里程碑一条记录）
- [x] 模块化的代码（必要时拆分为多文件）

### 进入阶段 3 的条件
> 所有计划内的系统都已就位并通过验收。游戏"功能上完整"——能玩完一局完整的流程。

---

## 阶段 3：打磨优化

### 目标
把"能玩"变成"想玩"。

### 和阶段 2 的区别

阶段 2 是加东西，阶段 3 是改东西。**阶段 3 不应该加新系统**——如果你发现还需要新系统，说明阶段 2 没完成，退回去。

### 打磨维度

按优先级排序（先做上面的，再做下面的）：

#### 1. 游戏手感（Game Feel）

这是打磨中回报率最高的部分。

```
[ ] 按键/触摸的响应延迟是否够低
[ ] 碰撞反馈——击中时有没有顿帧、震屏、粒子
[ ] 镜头是否跟随角色移动有微妙的滞后感（而非死死锁定）
[ ] 关键操作（跳跃、攻击、死亡）有没有音效反馈
[ ] 动作之间的衔接是否流畅（而非生硬切换）
```

**怎么和 AI 协作做手感？**

不要说"手感不好"——这太模糊。用对比描述：

```
好的反馈方式：
"跳跃落地太轻了，我希望落地时有一个 0.1 秒的微蹲 + 轻微震屏"
"攻击命中后角色和怪物都没反应，我希望有 0.05 秒的顿帧"
"移动启动太突然了，能不能加一个 0.1 秒的加速过程"
```

#### 2. 视觉美化

```
[ ] 色块替换为正式素材（可用 AI 生成图片）
[ ] 背景不是纯色（渐变、图案、或场景元素）
[ ] UI 有基本的设计感（对齐、间距、字号层级）
[ ] 关键时刻有视觉特效（得分、升级、死亡）
```

#### 3. 音频

```
[ ] 背景音乐（可用 AI 生成）
[ ] 核心操作音效（跳跃、攻击、收集）
[ ] 环境音效（如果适用）
[ ] 音量平衡（音效不会盖过音乐）
```

#### 4. 难度与平衡

```
[ ] 前 30 秒是否足够简单（让新玩家建立信心）
[ ] 难度曲线是否平滑（而非突然变难）
[ ] 最高难度是否有挑战但不绝望
[ ] 核心数值是否经过至少 3 次调整
```

#### 5. 用户体验

```
[ ] 新玩家第一次打开能否在 5 秒内理解怎么玩
[ ] 暂停、重新开始是否方便
[ ] 失败后的重试路径是否足够短（两次点击内回到游戏）
[ ] 加载时间是否可接受
```

### 打磨的工作节奏

每次和 AI 协作，只聚焦一个维度：

```
"这轮只做游戏手感。请读 docs/milestones.md 了解当前状态，
然后我们逐条过手感清单。"
```

不要一轮对话里同时改手感、换美术、加音效——改动太多无法判断是哪个改动引入了问题。

### 产出物
- [x] 打磨过的手感参数
- [x] 正式的美术素材和音效
- [x] 经过多次调整的难度曲线
- [x] 流畅的新手引导和 UX 流程

### 进入阶段 4 的条件
> 你自己愿意把这个游戏推荐给朋友玩。

---

## 阶段 4：发布上架

### 目标
准备发布素材，测试，提交 TapTap。

### 做法

#### 1. 准备发布素材

```
告诉 AI：
"游戏准备发布了。请帮我：
1. 生成游戏图标
2. 用游戏截图生成宣传图
3. 写一段 TapTap 商店描述"
```

需要的素材清单：
- [ ] 游戏图标（512x512）
- [ ] 至少 3 张游戏截图
- [ ] 1 张宣传横图
- [ ] 游戏描述文案

#### 2. 真机测试

```
"请生成测试二维码，我要在手机上测一下"
```

测试重点：
- [ ] 触控操作是否正常（特别注意手指遮挡）
- [ ] 不同屏幕比例下 UI 是否正常
- [ ] 性能是否流畅（特别是特效多的场景）
- [ ] 音频是否正常播放

#### 3. 发布

```
"素材都准备好了，请发布到 TapTap"
```

### 产出物
- [x] 完整的发布素材
- [x] 通过真机测试
- [x] 成功上架 TapTap

## 四个文档模板怎么用

| 模板 | 什么时候创建 | 什么时候更新 | 放在哪 |
|------|------------|------------|--------|
| `game-brief.md` | 阶段 1 开始时 | 很少更新（方向变了才改） | `docs/` |
| `decision-log.md` | 阶段 2 开始时 | 每做一个非显而易见的决策时 | `docs/` |
| `system-contract.md` | 每加一个系统前 | 系统接口变化时 | `docs/system-contracts/` |
| `milestones.md` | 阶段 2 开始时 | 每完成一个里程碑时 | `docs/` |

**不要预先创建所有文件。** 用到哪个创建哪个。阶段 1 只需要 `game-brief.md`。

## 和 AI 协作的关键原则

### 原则 1：每轮对话开头让 AI 读文档

```
"继续开发 XX 游戏。请先读：
- docs/game-brief.md（了解项目）
- docs/decision-log.md（了解已有决策）
- docs/milestones.md（了解当前进度）
然后我们继续做 [具体任务]。"
```

这一步不能省。AI 没有跨对话记忆，文档就是它的记忆。

### 原则 2：一次只做一件事

```
❌ "加上战斗系统、音效、还有把 UI 美化一下"
✅ "这轮加战斗系统。攻击、受击、死亡三个部分。"
```

### 原则 3：用预期清单验收

每次 AI 交付代码，要求它附上预期表现清单。你只需要逐条对照，用编号反馈。详见《预期清单驱动模式》。

### 原则 4：感觉不对就说，但要具体

```
❌ "感觉不太行"
❌ "移动怪怪的"
✅ "移动启动太快了，没有加速过程，像是瞬移"
✅ "跳跃落地后有一个明显的卡顿，像是掉了帧"
```

### 原则 5：重大方向变化，先讨论再动手

如果你想推翻一个早期决策（比如"还是不要 FSM 了，换行为树"），不要直接让 AI 改。先说：

```
"我在考虑把角色状态管理从 FSM 换成行为树。
请先评估一下影响范围——哪些代码需要改，风险在哪。
我们讨论完了再决定改不改。"
```

## 快速开始

1. 把 `templates/` 下的模板复制到你的项目 `docs/` 目录
2. 填写 `game-brief.md` 的三个必填项
3. 把 `AI-PROMPT.md` 的内容粘贴给 AI 作为协作指令
4. 开始阶段 1
