# 部署指南：三步启用 TapTap 制造工作流

> 总耗时：2 分钟。不需要任何技术知识。

---

## 这是什么

一套让 AI 自动管理游戏开发全流程的体系。部署后你只需要：

- 告诉 AI 你想做什么游戏
- 玩 AI 做出来的原型，说说感受
- 在 AI 给的选项里选一选

剩下的事——项目文件管理、进度追踪、质量检查、记忆维护、素材生成——AI 全部自动完成。

---

## 第一步：复制智能体准则

打开 `agent-guideline.md`，复制其中代码框内的全部内容。

粘贴到你项目的 `.claude/CLAUDE.md` 文件末尾。

> 如果你的项目还没有 `.claude/CLAUDE.md` 文件，创建一个，把内容粘贴进去。

## 第二步：复制 Skills

把 `skills/` 文件夹整个复制到你项目的 `.claude/skills/` 下。

完成后你的项目结构应该是：

```
你的项目/
├── .claude/
│   ├── CLAUDE.md                          ← 第一步粘贴的内容在这里
│   └── skills/
│       │
│       │── workflow-auto/                 ── 工作流 & 质量 ──
│       │   └── SKILL.md                   工作流自动驾驶
│       │── self-reflect/
│       │   └── SKILL.md                   代码质量复盘
│       │
│       │── memory-create/                 ── 记忆矩阵（5 个） ──
│       │   └── SKILL.md                   记忆创建/编码
│       │── memory-retrieve/
│       │   └── SKILL.md                   记忆检索/恢复
│       │── memory-consolidate/
│       │   └── SKILL.md                   记忆巩固/整理
│       │── memory-forget/
│       │   └── SKILL.md                   记忆遗忘/修剪
│       └── memory-distill/
│           └── SKILL.md                   记忆蒸馏/模式提取
│
├── scripts/
│   └── ...（你的游戏代码）
└── docs/
    └── ...（AI 会自动创建和管理这里的文件）
```

**7 个 Skill 的分工**：

| 类别 | Skill | 一句话说明 |
|------|-------|----------|
| 工作流 | `workflow-auto` | 自动判断项目阶段，执行对应的开发流程 |
| 质量 | `self-reflect` | 每次交付后检查代码质量、一致性、回归问题 |
| 记忆 | `memory-create` | 把重要决策和经验写入文档 |
| 记忆 | `memory-retrieve` | 新对话开始时从文档恢复项目上下文 |
| 记忆 | `memory-consolidate` | 定期整理记忆，合并重复，压缩索引 |
| 记忆 | `memory-forget` | 清理过时信息，防止记忆膨胀 |
| 记忆 | `memory-distill` | 从具体经验中提炼通用模式和用户偏好 |

## 第三步：开始做游戏

打开 TapTap 制造，对 AI 说：

```
"我想做一个游戏"
```

AI 会先让你选一个协作风格：

```
A) 🛠️ 游戏工程师伙伴 — 务实高效，做完就交付（默认）
B) 💡 创意碰撞师 — 主动给灵感，多聊创意
C) 🤫 沉默执行者 — 少说多做，只问关键问题
D) ✏️ 自定义 — 你来定义想要的助手风格
```

选完（或直接跳过用默认）之后，AI 就开始引导你做游戏了。

这个选择会保存下来，后续每次对话 AI 都会按你选的风格来沟通。随时可以修改 `docs/persona.md` 切换风格。

## 部署完成后会发生什么

| 你做的事 | AI 自动做的事 |
|---------|-------------|
| 第一次开始 | 让你选协作风格，保存偏好 |
| 说"想做 XX 游戏" | 创建设计简报，用选择题确认需求，做出原型 |
| 试玩后说感受 | 调整参数，重新交付 |
| 说"加个 XX 功能" | 检查冲突，写系统契约，实现功能，更新进度 |
| 什么都不说 | 给你建议"接下来可以做什么" |
| 说"发布" | 生成素材，测试，提交到 TapTap |
| 换了一个新对话窗口 | 自动从文档恢复全部记忆和人格设定，无缝衔接 |

你**不需要**：
- 手动创建任何文档（AI 自动创建）
- 告诉 AI "请先读 XX 文件"（AI 自动读取）
- 理解什么是"决策账本"或"系统契约"（AI 在后台维护）
- 提醒 AI 记住之前的决定（AI 通过记忆矩阵自动管理）
- 担心文档越来越多（AI 会自动整理、遗忘、压缩）

## 记忆系统是怎么工作的

你不需要理解这些细节，但如果你好奇的话：

**人怕遗忘，所以写日记写书。AI 也一样。**

这套系统模拟了人类记忆的三层结构：

```
┌─────────────────────────────────────────────────────┐
│  第一层：短期记忆（AI 的对话窗口）                       │
│  当前对话的所有内容。关掉对话就没了。                     │
├─────────────────────────────────────────────────────┤
│  第二层：长期记忆（docs/ 目录下的文件）                   │
│  项目索引、决策记录、系统契约、用户偏好……                 │
│  AI 每次新对话都会自动读取，恢复之前的工作状态。          │
├─────────────────────────────────────────────────────┤
│  第三层：外部记忆（docs/archive/ 目录）                  │
│  过时的决策、放弃的方案、旧版设计……                      │
│  就像人的日记本——平时不翻，但需要时翻得到。               │
│  AI 平时不读，但当你问"之前那个方案呢"时会自动去翻。      │
└─────────────────────────────────────────────────────┘

信息在三层之间流动：

  决策产生 ──→ 写入长期记忆 ──→ 过时后归档到外部记忆
                                         │
            需要时回流 ←─────────────────┘
```

5 个记忆 skill 各管一段：

| Skill | 做什么 | 人类类比 |
|-------|--------|---------|
| 创建 | 把重要的事记下来 | 上课做笔记 |
| 检索 | 想起相关的事 | 考试前翻笔记 |
| 巩固 | 整理归纳 | 期末复习总结 |
| 遗忘 | 不重要的收起来 | 旧笔记放书架 |
| 蒸馏 | 总结经验规律 | 毕业时写总结 |

**核心保证：AI 永远不会真的"忘记"任何东西。** 即使信息从长期记忆中移除了，也完整保存在外部记忆（archive）中，随时可以找回。

## 常见问题

**Q: 中途换了一个新的对话窗口，AI 还记得之前做的事吗？**

记得。AI 会在每次新对话开始时自动触发 `memory-retrieve` skill，读取 `docs/memory-index.md` 恢复项目状态。这就是这套体系的核心价值——把记忆从 AI 的脑子里搬到了文件里。

**Q: 文档文件会不会越来越多、越来越大？**

不会。`memory-consolidate` 负责压缩索引，`memory-forget` 负责归档过时信息，`memory-distill` 把具体经验提炼为简短模式。三者协作保证记忆始终精简。

**Q: 我能看 AI 自动创建的那些文档吗？**

当然可以。它们都在 `docs/` 目录下，是普通的 Markdown 文件。你随时可以看、可以改。

**Q: 如果我不喜欢 AI 给的建议怎么办？**

直接说"不要"或者"我想做别的"就行。AI 的所有建议都是选项，决定权在你。AI 还会把你的偏好记录下来，下次不再犯同样的错。

**Q: 我已经有一个做到一半的项目，能用这套工作流吗？**

可以。部署后告诉 AI "我有一个在做的项目，代码在 scripts/ 里，请先看看现在的状态"。AI 会读代码，自动创建 memory-index 和里程碑记录，然后继续。

**Q: 7 个 skill 会不会太多，影响 AI 响应速度？**

不会。Skill 只在满足触发条件时才加载，日常使用中通常只有 2-3 个 skill 同时活跃。而且 skill 之间有明确的职责划分，不会互相干扰。
