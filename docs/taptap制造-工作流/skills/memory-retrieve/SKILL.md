# Skill: 记忆检索（Memory Retrieve）

> 类比人脑：**线索触发回忆**。
> 人不会每次思考都把所有记忆翻一遍——而是当前情境作为线索，激活相关记忆。
> 想不起来时，还会翻日记本、翻笔记——这就是外部记忆回溯。
>
> 本 skill 负责三层记忆的读取：短期（当前对话）、长期（docs/）、外部（docs/archive/）。

## 触发条件

1. **新对话开始**（恢复项目上下文）
2. **即将实现新功能**（加载相关约束）
3. **用户提到了之前讨论过的内容**（按线索检索）
4. **遇到不确定的行为**（查证而非猜测）

## 检索策略：四级加载

**不要一次读所有文件。** 按需分级加载，控制上下文消耗。逐级递进，前一级找不到才往下一级找。

### 第一级：索引（每次对话必读，< 2K token）

```
必读：docs/memory-index.md
获取：项目名称、当前阶段、有效决策标题列表、活跃里程碑、下一步
```

这一级相当于人的**自传式记忆概要**——"我是谁、在做什么、做到哪了"。

### 第二级：相关详情（按任务需要读，每次 1-3 个文件）

根据当前任务，从索引中定位相关文件：

| 当前任务 | 需要读取的文件 |
|---------|--------------|
| 实现新功能 | 相关的 `system-contracts/`，可能冲突的 `decisions/` |
| 修 bug | `lessons.md`（有没有类似的已知问题） |
| 调整手感/数值 | `user-profile.md`（用户偏好） |
| 继续上次未完成的工作 | 标记为 🔧 的里程碑对应的契约 |
| 用户提到"之前说过的XX" | 在索引中搜索关键词，定位到具体文件 |

### 第三级：全量回顾（较少触发）

**只在以下情况读取全部长期记忆：**
- 用户要求"回顾整个项目"
- 即将进行重大架构变更
- 阶段转换时（如从阶段 2 进入阶段 3）

### 第四级：外部记忆回溯（翻日记本）

**当第一到三级都找不到答案时，自动翻阅 archive。**

```
触发条件（满足任一即回溯）：
1. 用户问"之前那个 XX"，但索引和详情中都找不到
2. 用户提到一个被取代的方案名称
3. 用户说"我们以前试过"/"之前不是做过"
4. 实现新功能时，发现需要参考一个曾被放弃的方案

回溯步骤：
1. 读取 docs/archive/archive-log.md（外部记忆目录）
2. 在目录表格中搜索关键词
3. 找到匹配条目 → 读取对应的 archive/ 文件
4. 找不到匹配 → 告知用户"我在所有记录中都没找到关于 XX 的内容"

回溯后处理：
- 如果找到的内容对当前任务仍然有用 → 触发"回流"（见 memory-forget 的回流机制）
- 如果只是回答用户的历史追问 → 回答后不回流，保持归档状态
```

**关键原则**：外部记忆回溯只在前三级无果时才触发，避免每次都扫描 archive 目录。archive-log.md 的表格结构让回溯成本很低——只读一个文件就能判断有没有。

## 线索匹配规则

当用户的消息包含以下线索时，自动检索对应记忆：

### 关键词线索

```
用户说"之前"/"上次"/"我们讨论过" 
→ 在 memory-index.md 中搜索匹配的决策或里程碑

用户说"为什么是这样的"/"当初怎么决定的"
→ 读取对应的 decisions/D-XXX.md，获取决策理由

用户说某个系统名称（"战斗"/"移动"/"UI"）
→ 读取对应的 system-contracts/XXX.md

用户说"又出这个问题了"/"之前也遇到过"
→ 读取 lessons.md，检索类似问题
```

### 冲突预检线索

```
用户要求的功能涉及已有系统
→ 自动读取该系统的契约，检查交叉场景是否有覆盖

用户要求改变之前的做法
→ 自动读取原决策，展示当初的理由和影响范围
```

## 检索失败处理

### 索引中找不到

```
如果用户提到的内容在 memory-index.md 中找不到：
1. 不要编造——直接告知"我没有找到关于 XX 的记录"
2. 问用户能否补充："你还记得大概是在做什么功能的时候说的吗？"
3. 如果用户能补充线索，再次检索
4. 如果仍然找不到，将此次信息作为新记忆编码（触发 memory-create）
```

### 索引文件不存在

```
如果 docs/memory-index.md 不存在：
1. 检查 docs/ 下是否有任何项目文档
   → 有：读取所有文档，重建索引（触发 memory-consolidate）
   → 没有：这是新项目，触发 workflow-auto 的阶段 1

2. 检查 scripts/ 下是否有代码
   → 有：读取代码，推断项目状态，创建索引
   → 没有：确认是新项目
```

## 上下文预算管理

每次检索后，估算已消耗的上下文：

| 已消耗上下文 | 策略 |
|-------------|------|
| < 20% | 正常，可以按需读更多文件 |
| 20-50% | 谨慎，只读必要的文件 |
| 50-70% | 紧缩，不再主动读取，只在必须时读 |
| > 70% | 告知用户"这轮对话上下文快满了，建议开一个新对话继续" |

**关键原则：宁可少读一个文件，也不要把上下文撑爆。** 索引的价值正在于此——它让你知道什么在哪里，而不需要把所有东西都加载进来。
