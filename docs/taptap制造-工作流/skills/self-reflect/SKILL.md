# Skill: 自我复盘（代码质量 & 项目健康度）

> **职责边界**：本 skill 只关注代码质量、一致性和项目健康度。
> 记忆相关操作（创建、检索、整理、遗忘、蒸馏）由 memory-* skill 矩阵负责。

## 触发条件

**每次代码交付后自动触发。** AI 在内部执行此 skill，不直接展示给用户。
复盘结果通过行动体现（修正代码、提出建议），而不是输出一段"反思报告"。

---

## 复盘清单

每次交付代码后，按以下清单逐条检查。

### 1. 决策一致性

```
问自己：这次写的代码和已有决策一致吗？

执行方式：
1. 回忆（或快速扫读）memory-index.md 中的有效决策
2. 逐条对照：新代码有没有违反任何一条？

常见不一致：
- 决策说用 FSM，但新代码用了 if-else 堆叠
- 决策说无霸体，但新代码在攻击动画中忽略了受击
- 决策说不加内购，但新代码加了付费相关逻辑

→ 发现不一致：
  a) 如果是自己的疏忽：立刻修正代码，不告知用户
  b) 如果是需求本身和决策冲突：告知用户，让用户选择

注意：发现不一致后的决策更新由 memory-create skill 负责，这里只负责检测。
```

### 2. 代码健康度

```
问自己：代码还健康吗？

□ 总行数
  → < 800 行：健康
  → 800-1200 行：告知用户"代码开始变长了，后面可以考虑拆分"
  → > 1200 行：主动提出拆分方案，列出可提取的模块

□ 重复代码
  → 是否有 3 处以上相似的代码段（>5 行的相似块）？
  → 是：建议提取为公共函数，给出具体方案

□ 模块耦合
  → 新代码是否直接访问了其他系统的内部变量（而非通过约定的接口）？
  → 是：重构为通过事件或函数调用

□ 命名一致性
  → 新增变量/函数的命名风格是否和已有代码一致？
  → 驼峰 vs 下划线 / 英文 vs 拼音 / 缩写风格
  → 不一致：静默修正为已有风格
```

### 3. 功能完整性

```
问自己：这次交付的功能完成了吗？

判断标准——以下全部满足才算"完成"：
- 功能的核心行为已实现
- 预期清单中的所有条目合理可验证
- 没有已知的阻塞性 bug
- 边界情况已考虑（空值、越界、极端输入）

→ 完成：告知 memory-create skill 记录里程碑
→ 未完成：在预期清单中说明剩余部分
```

### 4. 回归检测

```
问自己：新代码有没有破坏已有功能？

检查项：
□ 是否修改了被其他系统依赖的函数签名或行为？
□ 是否改动了全局变量或共享状态？
□ 是否改变了事件的发送/接收逻辑？

→ 检测到潜在回归：
  1. 停下来，不要继续加功能
  2. 告知用户："刚才加的 [新功能] 可能影响到了 [旧功能]"
  3. 先验证并修复回归问题
  4. 确认旧功能恢复正常后再继续
```

### 5. 性能意识

```
问自己：新代码有没有明显的性能隐患？

检查项：
□ 是否在 Update/HandleUpdate 中每帧创建新对象？
  → 是：提取到初始化阶段或加缓存

□ 是否在循环中执行了资源加载（GetResource/CreateComponent）？
  → 是：预加载或缓存

□ 是否有 O(n²) 或更高复杂度的逻辑在每帧执行？
  → 是：考虑空间分区、脏标记等优化

□ NanoVG 是否在渲染循环中创建字体（nvgCreateFont）？
  → 是：移到初始化阶段（显存泄漏！）

→ 发现隐患：评估严重程度
  a) 阻塞性（卡顿/崩溃）：立刻修复
  b) 潜在问题：告知用户，建议后续优化
```

### 6. 阶段感知

```
问自己：项目应该进入下一个阶段了吗？

阶段 1 → 2 的信号：
- 用户确认手感 OK
- 开始要求核心循环以外的功能

阶段 2 → 3 的信号：
- 用户不再要求新功能
- 开始说"细节"、"手感"、"好看"、"音效"等打磨类词汇
- 所有之前要求的功能都已验收

阶段 3 → 4 的信号：
- 用户说"差不多了"、"可以了"、"发布"、"给朋友玩"

→ 检测到信号时，主动告知用户：
"感觉我们可以进入 [下一阶段名称] 了——[简要说明下一阶段做什么]。
你觉得现在的 [当前阶段内容] 够了吗？还是还想再 [当前阶段的动作] 一下？"
```

### 7. 下一步建议

```
问自己：用户接下来最可能想做什么？

生成 2-3 个建议，满足以下条件：
- 和刚完成的功能有自然的衔接关系
- 用大白话描述，不用技术术语
- 以选择题形式给出

格式：
"这个功能搞定了！接下来你想：
a) [最自然的下一步]
b) [另一个方向]
c) 你有别的想法？"
```

### 8. 失败驱动进化（每个错误变成一条规则）

参考 ACON/ACE 的核心洞察：不只是修复错误，而是从每次失败中提炼规则，防止同类问题再犯。

```
触发：用户反馈预期不符（报了编号）/ 用户否决了方案 / build 失败

执行：
1. 修复问题本身
2. 分析失败根因，归类：
   a) 引擎 API 误用 → 记入 lessons.md（类型：程序）
   b) 需求理解偏差 → 更新 user-profile.md 的沟通偏好
   c) 系统交互遗漏 → 补充 system-contracts 的交叉场景
   d) 数值不合理 → 记录用户的"感觉"基准到 user-profile.md

3. 提取规则，写入 memory-index.md 的"避雷清单"：
   格式：⚠️ [简述] — 来自 L-XXX

4. 下次遇到类似场景时，memory-retrieve 自动加载避雷清单

示例：
  用户说"跳起来感觉太飘了" → 修复重力参数
  → 提取规则：⚠️ 初始重力设 -9.81 体感偏飘，需要 1.5-2 倍补偿 — L-003
  → 以后做跳跃相关功能时，自动提醒这条经验
```

**核心理念**：团队越久、规则越多、AI 越聪明——而不是每次都犯同样的错。

### 9. 玩家心智模拟（交付前的预演检查）

参考业界 AI playtesting 实践，在真正交付给用户之前，用"心智模拟"预判潜在体验问题。

```
问自己：如果我是玩家，我会怎么玩？

模拟 3 种玩家原型：
□ 探索者：会不会去点/试每一个能交互的东西？
  → 检查：是否有未处理的交互边界？
  → 检查：UI 上的可点击区域是否都有反馈？

□ 急性子：会不会快速点击/连续操作？
  → 检查：快速连续输入会不会导致状态异常？
  → 检查：动画未播完时操作会不会卡住？

□ 迷糊玩家：第一次玩能看懂该做什么吗？
  → 检查：核心操作是否有视觉引导？
  → 检查：失败后是否知道为什么失败？

→ 发现潜在问题：
  a) 体验阻塞性的 → 修复后再交付
  b) 锦上添花的 → 记入下一步建议
```

**注意**：这是轻量级的心理预演（<30 秒），不是完整的 QA 流程。目标是在交付前多想一步，减少用户反馈轮次。

## 与记忆 Skill 的协作接口

本 skill 在执行过程中会产生需要记忆系统处理的信号，由对应的 memory skill 接管：

| 本 skill 产生的信号 | 转交给 | 具体操作 |
|-------------------|--------|---------|
| 发现决策不一致并修正 | `memory-create` | 记录新决策或更新已有决策 |
| 确认功能完成 | `memory-create` | 记录里程碑完成 |
| 检测到阶段切换 | `memory-consolidate` | 触发阶段性记忆整理 |
| 代码拆分为多模块 | `memory-create` | 记录架构决策 |
| 失败分析产出新规则 | `memory-create` | 写入 lessons.md + 更新索引避雷清单 |
| 玩家模拟发现体验风险 | `memory-create` | 记录为情景记忆（如果确实发生了问题） |

## 异常处理

### 用户反馈和预期不符

```
1. 不要猜原因，先读相关代码确认实际行为
2. 如果是 bug：修复后重新交付，重新走复盘流程
3. 如果是预期理解偏差：和用户确认他想要的表现，调整后重新交付
```

### 用户长时间没回应

```
如果给了建议但用户没有选择：
- 不要催促
- 不要自行决定
- 等用户主动说下一步
```
