# Skill: 记忆遗忘（Memory Forget）

> 类比人脑：**人怕遗忘，所以写日记、写书。**
> 遗忘不是丢弃，而是把信息从"随身携带"降级到"书架上可查"。
> archive/ 目录就是 AI 的日记本——不随身翻，但什么时候需要都翻得到。
>
> 本 skill 负责管理**长期记忆 → 外部记忆**的降级流动（见 agent-guideline 三层记忆模型）。

## 触发条件

1. **memory-index.md 超过 60 行**（容量预警）
2. **单次检索需要读取 5 个以上文件**（检索效率下降）
3. **memory-consolidate 完成后发现仍然臃肿**（巩固不够，需要修剪）
4. **阶段转换时**（进入新阶段，旧阶段的细节可以淡化）
5. **项目重大方向调整**（旧方向的决策大面积失效）

## 遗忘不是删除

**关键区分：**

| 操作 | 含义 | 实际动作 |
|------|------|---------|
| 遗忘 | 从索引中移除，不再主动加载 | 移入 `docs/archive/` |
| 删除 | 永久销毁 | **永远不做** |

`docs/archive/` 就是项目的"日记本"——完整记录了每一条被降级的决策、每一版被替换的方案、每一个走过的弯路。AI 日常不翻阅这本日记，但当用户追问历史、当前长期记忆检索不到结果、或需要回溯旧方案时，`memory-retrieve` 会自动来这里查找（见 memory-retrieve 的"外部记忆回溯"章节）。

## 三种遗忘机制

### 机制 1：衰减遗忘（时间驱动 × 重要性分级）

衰减速度由 `memory-create` 标注的**重要性评分**决定：

| 重要性 | 衰减窗口 | 含义 |
|--------|---------|------|
| ★★★ | **永不衰减** | 核心玩法/架构/用户红线 |
| ★★☆ | 5 个里程碑未引用 → 休眠，再 3 个 → 归档 | 跨系统关键决策 |
| ★☆☆ | 3 个里程碑未引用 → 休眠，再 2 个 → 归档 | 局部参数、备选方案 |

```
执行：
1. 扫描 memory-index.md 中的所有有效决策
2. 读取每条决策的重要性评分（★★★ / ★★☆ / ★☆☆）
3. 根据上表的衰减窗口判断是否降级
4. ★★★ 直接跳过——无论多久未引用都保留

示例：
D-003 ★★★ "跳跃高度 3 米" — 核心玩法参数
→ 永不衰减，始终保留

D-007 ★★☆ "UI 用 Flexbox 布局" — 跨系统架构决策
→ M-001 创建，M-006 仍未引用（5 个里程碑）→ 降级"休眠"
→ 但下次做 UI 相关任务时一定会引用，大概率不会真的归档

D-008 ★☆☆ "考虑过用粒子系统做雨效果" — 备选方案
→ M-003 创建，M-006 仍未引用（3 个里程碑）→ 降级"休眠"
→ 再 2 个里程碑仍未引用 → 归档
```

**衰减例外清单（即使评分不是 ★★★ 也不衰减）：**
- 用户明确表达的"绝对不要"（红线决策）
- 避雷清单中的条目（踩过的坑不能忘）

### 机制 2：替代遗忘（冲突驱动）

```
规则：当新决策明确取代旧决策时 → 旧决策立即归档

执行：
1. 新决策创建时（memory-create），检查是否存在同主题的旧决策
2. 如果新决策的"选择"和旧决策不同 → 旧决策归档
3. 在归档记录中标注"被 D-XXX 取代"

示例：
D-002 "用 FSM 管理角色状态"
D-019 "改用行为树管理角色状态"（明确取代 D-002）
→ D-002 归档，标注"被 D-019 取代"
→ 索引中只保留 D-019
```

### 机制 3：压缩遗忘（精度降级）

```
规则：不需要完整详情的记忆 → 压缩为一行摘要

执行：
1. 对于"值得知道存在，但不需要完整内容"的记忆
2. 将详情文件内容压缩为索引中的一行摘要
3. 详情文件移入 archive/（保留但不主动读取）

示例：
原始（10 行决策文件）：
  D-005: 受击硬直时长
  选择：0.3秒  备选：0.2/0.5秒
  理由：平衡打击感和流畅度
  影响：连招系统、无敌帧时长
  ...

压缩后（索引中的一行）：
  D-005: 受击硬直 0.3s（平衡向）
  
完整文件保留在 archive/D-005.md
```

## 遗忘执行流程

```
1. 触发遗忘 skill

2. 读取 memory-index.md，统计：
   - 当前有效决策数量
   - 当前休眠决策数量
   - 总行数

3. 按优先级执行遗忘：
   a) 替代遗忘：先处理明确被取代的旧决策
   b) 衰减遗忘：再处理长期未引用的决策
   c) 压缩遗忘：最后对仍在索引中但不需要详情的条目做压缩

4. 执行归档操作：
   - 将归档的文件移入 docs/archive/
   - 更新 memory-index.md
   - 在索引的"已归档"区添加一行摘要

5. 验证瘦身结果：
   - memory-index.md 是否 < 60 行？
   → 是：完成
   → 否：检查是否有更多可归档的内容，或提议更激进的压缩
```

## 外部记忆（archive）目录结构

```
docs/archive/                      ← 项目的"日记本"
├── decisions/                     # 曾经的决策
│   ├── D-002.md                   # 标注"被 D-019 取代"
│   └── D-008.md                   # 标注"备选方案，未采用"
├── contracts/                     # 曾经的系统契约（系统被重构后）
├── snapshots/                     # 阶段性快照（阶段切换时自动生成）
│   └── phase-1-summary.md         # 原型阶段的完整总结
└── archive-log.md                 # 📖 外部记忆目录索引（核心文件）
```

### archive-log.md 格式

archive-log.md 不只是"什么时候删了什么"的日志，而是外部记忆的**目录索引**——memory-retrieve 回溯时首先读这个文件来定位需要的内容。

```markdown
# 外部记忆目录

> 这里记录了所有从长期记忆降级到外部记忆的内容。
> 每条记录包含：原始编号、一句话摘要、归档原因、文件位置。
> 当需要回溯时，先在这里找到方向，再去读具体文件。

## 决策归档
| 编号 | 摘要 | 归档原因 | 归档时间 | 位置 |
|------|------|---------|---------|------|
| D-002 | FSM 管理角色状态 | 被 D-019 行为树取代 | M-008 后 | decisions/D-002.md |
| D-008 | 粒子系统做雨效果 | 备选方案未采用 | M-005 后 | decisions/D-008.md |
| D-005 | 受击硬直 0.3s | 已压缩为摘要，详情归档 | M-006 后 | decisions/D-005.md |

## 契约归档
| 系统 | 归档原因 | 归档时间 | 位置 |
|------|---------|---------|------|
| old-combat | 战斗系统重构 | M-012 后 | contracts/old-combat.md |

## 阶段快照
| 阶段 | 摘要 | 归档时间 | 位置 |
|------|------|---------|------|
| 阶段 1 | 原型冲刺：确定跑酷核心，3 个 M 完成 | 阶段切换时 | snapshots/phase-1-summary.md |
```

**关键设计**：archive-log 的表格让 memory-retrieve 只需读一个文件就能判断归档中有没有需要的东西，而不用扫描整个 archive/ 目录。

## 阶段快照（阶段切换时自动生成）

每次项目从一个阶段切换到下一个阶段时，自动生成该阶段的完整快照：

```
文件：docs/archive/snapshots/phase-[N]-summary.md

内容：
# 阶段 [N]：[阶段名称] 总结

## 时间跨度
从 M-001 到 M-XXX，共 N 个里程碑

## 核心成果
- [完成了什么]

## 关键决策
- D-XXX: [仍然有效的决策摘要]
- D-XXX: [已被取代的决策摘要]

## 踩过的坑
- [教训 1]
- [教训 2]

## 用户偏好发现
- [这个阶段中了解到的用户偏好]
```

**价值**：当项目复杂度增长后，阶段快照让 AI 能快速了解"这个项目经历了什么"，而不需要读每一条决策的细节。

## 外部记忆回流（archive → 长期记忆）

遗忘不是单向的。当归档的内容再次变得相关时，可以回流到长期记忆：

```
触发条件：
1. 用户主动问："之前那个 XX 方案是怎么回事？"
2. memory-retrieve 在长期记忆中找不到答案，自动回溯到 archive
3. 项目方向调整，旧方案重新变得可行

回流操作：
1. 从 archive/ 中找到对应文件
2. 重新添加到 memory-index.md 的有效决策中
3. 在 archive-log.md 中标注"已回流"
4. 保留 archive 中的副本（不移除，作为历史记录）
```

## 安全规则

1. **永远不删除文件。** 归档 = 降级到外部记忆，不是销毁
2. **核心决策不遗忘。** 核心玩法、架构选择、用户"绝对不要"永远保留在长期记忆
3. **遗忘操作可逆。** 外部记忆随时可以回流到长期记忆
4. **遗忘过程不告知用户。** 静默执行，用户感知到的是"AI 的回答更聚焦了"
5. **archive-log 必须更新。** 每次归档都要更新 archive-log.md 的目录表格，这是回溯的唯一入口
